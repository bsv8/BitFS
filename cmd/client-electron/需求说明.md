# BitFS 客户端 Electron 壳需求说明

## 1. 目标

这个项目为现有 BitFS 客户端增加一个 Electron 桌面壳。它的外观和交互像浏览器，但它不是互联网浏览器，而是一个只信任 `bitfs://` 的内部资源浏览器。系统内所有文件身份都以 `seed_hash` 为准，页面、脚本、样式、图片、音视频和直播资源都只能按 hash 获取。

## 2. 信任边界

- `bitfs://<seed_hash>` 是应用内唯一可信内容世界。
- `http://`、`https://`、域名跳转和传统 URL 对这个应用都属于不可信外部世界，必须禁止访问。
- 页面拥有普通浏览器的 JS/CSS 执行能力，并按产品决策开放 Electron renderer 的 Node 能力。
- 同时保留 `window.bitfs` 统一入口，给页面使用客户端能力，避免页面直接耦合主进程细节。

## 3. 资源装载原则

- 主页面通过 `bitfs://<seed_hash>` 打开。
- HTML 内部引用的子资源如果最终解析结果是合法 hash，就允许继续装载。
- 非 hash 资源一律拒绝，不进入下载、不进入预算许可流程。
- 资源装载前优先检查本地是否已经命中；本地已存在时直接使用，不触发预算判断。

## 4. 预算与许可

静态资源使用两层自动装载阀值：

- 单个静态资源自动装载阀值
- 单个页面累计自动装载阀值

直播资源使用独立预算阀值，按直播消费模型单独判断，不与静态资源共用。

当静态资源未本地命中时，系统先做资源计划，计算该资源的预估总价。静态文件统一按 `64K` 分块均价计算，所以价格判断可以稳定建立在 `seed_price + chunk_price * chunk_count` 上。低于静态阀值的资源自动装载；超过阀值的资源先挂起，再由用户许可后继续。

## 5. 浏览链路

用户输入一个 `seed_hash` 打开主页面后，Electron 壳负责完成以下流程：

1. 打开主文档。
2. 解析主文档中继续引用的 hash 子资源。
3. 对未本地命中的静态资源发起批量资源计划。
4. 按单资源阀值和页面累计阀值决定自动放行或挂起。
5. 当资源被挂起时，在浏览器壳顶部显示预算许可条，让用户决定是否继续。
6. 用户许可后，资源继续走真实下载链路并回填到页面。

这个流程的重点不是把页面渲染出来，而是让“浏览动作”和“真实链上付费获取资源”保持一致。

## 6. 与现有客户端的关系

Electron 壳不重做 BitFS 业务。现有 Go 客户端仍然是业务内核，负责：

- 真实发布需求
- 等待卖家客户端报价
- 选择报价并完成真实下载
- 维护本地缓存和索引
- 管理直播链路与费用池

Electron 壳负责：

- `bitfs://` 协议入口
- 浏览器式窗口、地址栏与导航控制
- 预算许可条
- 页面 Node 能力与 `window.bitfs` 能力桥
- 把页面请求接回客户端的 `get-file` 业务域
